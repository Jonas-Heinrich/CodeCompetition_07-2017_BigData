Introduction

Big Data is on everyone's lips. Data is knowledge and knowledge is power. But what is the point behind this saying and the buzzword "Big Data"?
That is what we want to deal with in this Code Competition. "Big Data" is about the evaluation of large amounts of data. The evaluation firstly consists of processing, so that it can be automatically analyzed by appropriate algorithms. Then it is about the recognition of patterns and connections within the data to ultimately derive knowledge and visualize it.
The following steps are therefore important:

  • Processing
  • Pattern Recognition
  • Knowledge derivation
  • Visualize the results

Input

We provide you with data from a racing game. In this record you will find 160,000 races.
Out of this (meta)data you can already make interesting statements (Who drives when preferably and against whom races? Won or lost?) and derive predictions.
Tip: Filter the entries based on the questions you ask it.

About the Data

The data contains the following information/columns:

  • ID: The ID of the Entry
  • race_created: When was the race planned?
  • race_driven: When did the race take place?
  • track_id: ID of the racetrack where the race took place.
  • challenger: userID of the challenger
  • opponent: userID of the enemy (challenged)
  • money: Bet on the race (in EUR)
  • fuel_consumption: Fuel consumption of the participants during the race (in l)
  • winner: Winner of the race (userID)
  • status: The status of the race. Possible are: waiting, finished, retired, declined

  • forecast: Weather forecast for the race.
    a:4:{s:5:"sunny";i:10;s:5:"rainy";i:70;s:8:"thundery";i:0;s:5:"snowy";i:20;}
    The a:4 indicates a serialized array with 4 data pairs. The content of the array is between the curly braces. Every data pair represents a kind of weather and the probability of its occurrence. The kinds of weather (entries in the array) are separated by semi colons. There are 4 types of weather (sunny rainy, thundery, snowy). In front of every kind there is an „s:“ and an integer. The integer describes the length of the following string (e.g. sunny 5 characters, thundery 8 and so on). Behind the type of weather there is also an „i:“ and another integer. The number behind the i (i for integer) describes the likelihood of the weather happening in percent.

  • weather: Actual weather

This graph shows the typical lifecycle of a video game. At release it starts off very strong, but as time goes on it loses popularity.
With the introduction of new content (such as DLC, add-ons, etc.) or a sale, player numbers climb in the short term, leading to spikes in activity. After a year the game fades into irrelevance as new titles are released and developers as well as players move on to the next title.

Conclusion

The range of the amount of fuel that is needed to complete the race varies significantly. Given that distance is roughly proportional to the distance travelled, one can deduct that track number 12 is neither very long nor short. This might be a factor that increases its popularity.

The Data

For specifics on the data please review About this competition.

Main.py

This file contains the high level logic of the processing script. It coordinates the processing, and accumulates the results of the respective parts in a variable.

It then saves these results in two files to the JavaScript solution directory. There are two files which are split because of the time it takes to load the player data. DataCollection.js contains the data for the sections The Raw Data, General Data, Track Data and Weather Data. PlayerDataCollection.js contains all Player Data.

DataItem.py

This file contains the definition for the DataItem class. It is used to store all the data for every race in races.csv.

It takes a line as a parameter and parses it to extract the necessary data. The data can then be accessed by calling its properties. It also has methods to simplify conditionals.

Parse.py

This file contains the code that handles the parsing of races.csv. The GetInput() method returns an array of DataItem(s).

Output.py

This file contains useful functions for generating json strings from an object, specifically a list or a dictionary.

PlayerItem.py

This file contains the definition of the PlayerItem class. It is used to store information about a player.

It takes a player id as a parameter. Through public properties and a variety of methods it is possible to manipulate data about a player. This data can then be turned into a json string.

Processing.py

This file contains the core of the application. It contains functions for every data segment of the website. Each function takes the parsed data as an input, further processes it and returns a JSON string.

JavaScript - Behaviour

index.js

This file contains the main behaviour of the website. Any button clicks on the website lead to a function that is called in index.js.

animation.js

Animation.js contains various functions for animating and manipulating html elements. It does not act itself, rather is it mainly called from outside to manipulate.

Language.js

Language.js contains the logic that is needed to display the correct language.

There are 2 types of language resources. The LanguageEntries and the LanguageRepository. The former contains an array of LanguageEntry(s), which are used to automatically update the language of the website by saving its class name / id and the piece of text associated with it.
The LanguageRepository is an object which is used by other pieces of code to get a certain string that is identified by another unique string identifier.

JavaScript - Solution

main.js

main.js contains the top level logic of the solution. It is responsible for calling the processing methods and finishing the loading animations once the data is loaded.

DataCollection.js

DataCollection.js is generated by the Python code and contains all the relevant data for the Raw Data, General Data, Track Data and Weather Data section. It contains a JavaScript object that can be accessed by the other pieces of code in a structured way.

PlayerDataCollection.js

PlayerDataCollection.js is generated by the Python code as well and contains all the relevant data for the Player Data section. It expands the JavaScript object DataCollection.

PlayerDataInterpreter.js

PlayerDataInterpreter.js is the interface that the other code uses to access the player data section in the DataCollection object.
The PlayerData is generated as an unstructured array in order to save space. In order to make programming easier, the Interpreter provides methods that accesses certain data in the array and returns it parsed and ready to use.
An important note is that the player id and the index in the player data array do not correlate. Some player ids do not show up in the dataset and would therefore be wasted space. Whenever you read playerIndex (or just index), it means the actual index in the array and playerID (or just id) means an actual playerID that might not even exist in the data.

Output.js

Output.js contains various functions that are used by other code to display the data on the screen. It can build image collections, placements, tables and charts.

Processing.js

Processing.js is the core of the website. It takes the data from the DataCollection object and displays it in a meaningful way.
Each section has its own dedicated distributor function that in turn has more functions for every solution section. The functions are executed in async and are called by main.js when the data has loaded.
